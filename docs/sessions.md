# Sessions & Memorization

## What Is a Session?

A **session** is a single continuous conversation. Each session has:
- A UUID identifier
- A start timestamp (`startedAt`)
- An optional end timestamp (`endedAt`, `null` while active)
- Provider and model metadata
- The full message array

Sessions are stored as JSON files in the `logs/` directory (auto-created, git-ignored) and are named `<uuid>.json`.

### Lifecycle Events

| Event | What happens |
|---|---|
| **App start** | Settings and history loaded from `localStorage`. If the last message was ‚â• 3 hours ago, the old session is finalized silently and a new one starts. |
| **Message sent** | `lastMessage` timestamp updated; 3-hour inactivity timer resets. Session written to server (fire-and-forget). |
| **3-hour idle timeout** | Session stamped with `endedAt` and saved. New session starts automatically. Memorization begins in the background. |
| **Manual clear** | **Clear** button closes and memorizes the current session before starting a fresh one. |

### Session ID and Timestamps

- `sessionId` ‚Äî UUID generated by `crypto.randomUUID()` (or a polyfill fallback).
- `sessionStartedAt` ‚Äî ISO timestamp set when the session is created.
- `lastMessage` ‚Äî ISO timestamp of the most recent message, persisted in `localStorage` to survive page reloads.

```json
  "updatedAt":  "2026-05-11T17:12:05.000Z",
  "messages": [
    { "role": "user",      "content": "Hello", "timestamp": "2026-05-11T14:30:05.000Z" },
    { "role": "assistant", "content": "Hi!",   "timestamp": "2026-05-11T14:30:07.341Z" }

```

Tool-call turns (role `tool` and assistant turns containing `tool_calls`) are included in the session log but are stripped when exporting to Markdown.

## Session Memorization

When a session closes ‚Äî either by the 3-hour idle timeout or by manually clearing the chat ‚Äî the full conversation is automatically sent to the configured LLM. The model extracts distinct topics discussed and returns structured JSON. Each topic becomes a new [Lorebook](lorebook.md) entry.

### What Gets Saved Per Entry

| Field | Content |
|---|---|
| Comment (title) | A concise label for the topic |
| Content | A third-person narrative summary of key facts, decisions, and details worth remembering in future conversations |
| Keys (keywords) | 3‚Äì8 trigger keywords for the entry |
| Position | `before_char` (default for memorized entries) |

### Limits and Conditions

| Condition | Behaviour |
|---|---|
| Session has fewer than 4 readable messages | Skipped ‚Äî too short to summarize meaningfully |
| No API key configured | Silently skipped |
| 1‚Äì8 entries per session | The LLM is asked to return between 1 and 8 topic entries |
| Background execution | Runs after the new session has already started; never blocks the UI |
| Race-condition safety | The lorebook is fetched fresh from the server before writing, so no concurrent edits are lost |

### User Feedback

A brief on-screen toast confirms the result, e.g.:

> *"3 lorebook entries memorized from the last session."*

### Editing Memorized Entries

Entries created by memorization are indistinguishable from hand-crafted lorebook entries and can be edited, disabled, or deleted at any time via **‚ò∞ ‚Üí Lorebook ‚Üí View entries**.
