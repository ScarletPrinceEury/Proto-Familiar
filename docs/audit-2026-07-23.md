# Codebase audit — 2026-07-23

A full-repo audit pass (requested by the ward): end-to-end wiring, orphaned/stale
code, spec conformance, comment hygiene, and scientific-consensus checks. Test
baseline at audit time: **all green** — 1,583 JS tests, 311 Unruh, 138 Phylactery.

Fixed in this same commit (mechanical, restoring documented intent — no
safety-path behavior changed):

- **`thalamus.js getMemoriesBySubject` referenced `phylacteryClient`, a variable
  that does not exist** (the module's client is `mcpClient`). Every call threw a
  `ReferenceError`, silently swallowed by discord-gateway's
  `.catch(() => ({ items: [] }))` — so the villager `!consent` menu (0.8.106,
  "what the Familiar holds about THEM") could structurally never show kept
  memories (`memCount` was always 0). Exactly the swallowed-ReferenceError class
  the 0.9 post-mortem wrote rules about; no pipeline test covered this path.
- **Double `tickSurfacing()` in the non-streaming `/api/chat` loop path** —
  called once bare and once with tool names per turn. `tickSticky` decays TTLs
  on every call, so sticky tool modules expired twice as fast on non-streaming
  turns. The bare call is removed.
- Dead `store` variable in `POST /api/gcal/google/credentials`; two indentation
  glitches in server.js.
- `docs/architecture.md`'s "Autonomous loops" table was missing the **Noticing**
  loop entirely (documented elsewhere in the same file) — added.

## Findings needing ward sign-off (NOT fixed here — safety-critical paths)

### 1. crisis-signals.js: severe patterns self-damp on their own wording

The severe pattern `\b(don'?t|do not) want to (be here|exist|live)\b` contains
"don't" — which is also a `NEGATION_BLOCKERS` entry, and the damping context
window (±50 chars) **includes the matched text itself**. Result, verified live:

| Message | Expected | Actual |
|---|---|---|
| "I don't want to be here anymore" | severe, weight 8 | damped to **1.6** |
| "I don't feel like myself" | moderate, 2 | damped to **0.4** |
| "it doesn't feel real" | moderate, 2 | 2 (not damped — "doesn't" isn't a blocker; inconsistent with the above) |

The header comment already documents this exact hazard for "can't" ("they appear
inside many valid distress signals… treating them as negation damps every other
distress signal") — the same reasoning was never applied to "don't", which
appears *inside a severe signal's own pattern*. The existing test masks it:
`assertScored(..., { levelMin: 1 })` passes at 1.6.

At weight 1.6, a single "I don't want to be here anymore" lands the tier at
**calm→mild** instead of severe→triage. This is the 1.5-hour-silence failure
class expressed in code. Suggested shape of a fix (for sign-off): exclude the
match's own span from the damping context, or per-signal opt-outs for blockers
that are part of the signal's grammar — plus tightening the masked test to
assert full weight.

### 2. UTC window bounds passed into Unruh's local-naive schedule queries

Unruh stores and compares **ward-local-naive** timestamps, and
`schedule.get_window` does *not* run `to_local_naive` on caller-supplied
`from_ts`/`to_ts` (they go straight into SQL `BETWEEN` string comparison).
Several Node call sites pass UTC `new Date(...).toISOString()` bounds:

- `server.js gatherNoticingWakeInputs` (`winFrom`/`winTo` — feeds the noticing
  wake conditions, incl. overdue-event and readiness scans);
- `cerebellum.js decideTriageViaLLM` candidate-tasks lookup (`fromIso`/`toIso`).

On a server whose zone differs from the ward's (the WSL/Docker/hosted case
0.7.86 exists for), these windows shift by the UTC offset — up to ±14h. The
correct helper (`wardLocalShiftedISO` / ward-local-naive formatting) already
exists and is used by `/api/temporal/schedule` and the reflection assembly.
Because noticing and triage are ward-sign-off surfaces, flagged rather than
fixed. Alternative robust fix: normalise `from_ts`/`to_ts` through
`to_local_naive` inside `get_window` itself (one seam, all callers healed) —
note that offset-bearing bounds would then convert using the *server's* TZ env,
which Unruh already receives as the ward's zone at spawn.

### 3. `visibleAudiences`: equal-score circles see each other's records

The Pillar E recall gate ranks categories by scalar `permissionScore` and lets a
room see every category scoring **≤** its own. Two unrelated circles configured
with identical grants (say "Family" and "Work colleagues", both
`memories:'shared'`) therefore see records tagged for *each other*. The scalar
is a proxy for trust, but circle membership is not a total order — a fact
disclosed to Family is not thereby cleared for Work. The per-topic content gate
(Phase 4) narrows *what kind* of content, not *which circle*. Worth a deliberate
decision: either intended ("equal trust = equal visibility") and documented, or
the gate should require tag ∈ {room's own tag + strictly-lower tiers} /
an explicit circle-inclusion relation.

## Other findings (non-safety; recommended follow-ups)

1. **`enqueueSessionByDay` / `memorize_now` respects coverage, good — but the
   villager-consent memory list now works again** (the fix above). Worth a
   pipeline test: `consentStateFor` → real thalamus call with a stubbed MCP
   client, so the next renamed variable can't silently zero the menu again.
2. **`/api/chat` rate-limiter map is unbounded** (`_chatRateCounts` never
   prunes old IPs). Harmless on localhost; a slow leak behind Tailscale with
   many client IPs. A periodic sweep or LRU cap would close it.
3. **`docs/architecture.md` reminders-loop row** doesn't mention the weather
   refresh + elapsed-stamping passes that ride the same tick (both documented
   elsewhere; the table row is the discoverable summary).
4. **Ward Discord DM turns can end in silence** when a tool chain produces no
   closing text *without* hitting the round cap (discord-gateway `handleTurn`:
   "stayed quiet" branch). For ambient rooms that's right; for a ward's direct
   question it's dead air the round-cap machinery was built to prevent. Worth
   a ward-turn-only nudge (one forced text round, same as cap exhaustion).
5. **Repo-root `Research/` and `wiki/` mix** working notes with the shipped
   tree; `Research/SampleLB12122025.json` looks like a stray data sample.
   Consider moving genuinely-load-bearing notes under `docs/research/` and
   pruning the rest.
6. **codealmanac isn't runnable in ephemeral CI/cloud sessions** (not
   installed, and its Claude-agent harness isn't available), so the CLAUDE.md
   "every session" almanac rule silently no-ops there. Worth a note in
   CLAUDE.md scoping the rule to environments where the tool exists.

## What the audit did NOT find

- **No orphaned modules.** Every root JS file is imported (server.js is the
  entry). The many exports with no production consumers are the deliberate
  export-for-test pattern, each with test coverage.
- **No copy-paste duplication of substantial logic** was found; shared helpers
  (slug-ids, llm-call, message-sanitize, macros, audience) are consistently
  reused. RULE A (callProviderChat / 4000-cap + extractContent) holds at every
  LLM call site checked, including both Discord reply boundaries.
- **Comment density is high but overwhelmingly load-bearing** (rationale,
  invariants, recorded incidents) — consistent with the repo's explicit
  institutional-memory convention. A small amount of changelog-in-comments
  ("was missing in v1", version stamps) could migrate to git history, but
  removing it would trade against the repo's stated philosophy; left alone.
- **Graceful-degradation rule holds** everywhere checked: no module can throw
  into the chat path; every loop has its env off-switch; adapters fail
  independently.
- **Science checks:** the crisis-resource numbers (988, Samaritans 116 123,
  findahelpline) are current; the BA/PHQ-9/EMA-burden claims in
  trackers-research.md are consistent with the literature; CLAUDE.md's
  prompting-research claims are already self-calibrated (its own point 6). The
  one consensus-adjacent caveat: pure lexicon/regex crisis detection is known
  to have both poor sensitivity and specificity relative to contextual models —
  the file says so itself, and `flag_distress` (the model's own read) is the
  right complement. Finding 1 above matters precisely because the regex tier
  is the *first* line and currently under-weights a canonical severe phrasing.
